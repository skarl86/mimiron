## §B — Issue #21 (J5 apply-check dimension)

### What shipped

- `skills/bench-judge/SKILL.md` — 5-dim rubric (J5 added), formula `(J1+J2+J3+J4+J5)/5`, deterministic J5 step ordered *before* LLM judging so J5 is captured even if LLM phase fails.
- J5 lightweight default = `git apply --check` against `base_ref`. Strong opt-in mode = `apply` + `py_compile` on changed `.py`. Known limitation (runtime NameError invisible to py_compile) documented inline + cross-referenced to #20.
- Optional JSON field `apply_check: 0.0 | 1.0`. Absent → backward-compat fallback to 4-dim mean.
- Re-judge of B01/B02/B03 dogfood-005 candidates — retrospective projection (J1-J4 preserved, J5 added deterministically).

### Re-score table (dogfood-005 candidates, new rubric)

| Bench | J1 | J2 | J3 | J4 | J5 | new sim | old sim | new verdict | old verdict | Δ |
|---|---|---|---|---|---|---|---|---|---|---|
| B01-welcome-message-fix | 0.50 | 0.10 | 0.30 | 0.15 | **1.00** | 0.41 | 0.27 | **passed** | failed | ⚠ regression |
| B02-counselor-handover | 0.85 | 0.25 | 0.30 | 0.20 | **0.00** | 0.32 | 0.40 | **failed** | passed | ✓ correction |
| B03-product-resolver-refactor | 0.95 | 0.90 | 0.95 | 0.95 | **1.00** | 0.95 | 0.94 | passed | passed | — |

Verdicts computed via `bench_score = 0.6 * test_pass_rate + 0.4 * sim`, cutoff 0.75, `test_pass_rate = 1.0` (target_ref worktree, current `runner` behavior).

### Observations

- **B02 catastrophic catch worked** — the dogfood-005 candidate has hand-edited hunk counts (`@@ -125,6 +140,32 @@` is broken), so `git apply --check` failed with "corrupt patch at line 142". J5=0.0 dragged the score down, verdict flipped to `failed`. This is the desired outcome.
- **B02's NameError is NOT what J5 caught** — the candidate calls a fictitious `daily_tracker.mark_counselor_active`, but that's a *runtime* error that even `py_compile` strong-mode wouldn't catch. J5 happened to fail for a *different* surface reason (corrupt hunk metadata). Still catastrophic, still correctly rejected, but the catch is incidental.
- **B01 regression** — the wild-guess candidate now scores 0.41 (vs old 0.27) because J5=1.0 averages in. bench_score 0.764 > 0.75 cutoff → flips to `passed`. This is a *real* regression introduced by J5 alone. **It will be re-caught by #20** (sim < 0.5 force-fail gate) — B01's new sim is 0.41 < 0.5.
- **B03 unchanged** — the refactor candidate already applies cleanly, J5 just nudges score up marginally.
- **Retrospective projection vs fresh judge call** — we did NOT re-invoke the bench-judge skill. Instead we kept the existing J1-J4 samples and projected J5 onto each. Justification: J5 is deterministic (binary, depends only on the candidate diff and base_ref), so calling LLM 9 times would only add noise to J1-J4 and confuse the experiment. The methodology is documented in each judge JSON's rationale field.

## §C — Issue #23 (4-label certainty band)

### What shipped

- `skills/bench-judge/SKILL.md` §"흐름" step 6 — 4 mutually-exclusive labels:
  - `uncertain` (spread ≥ 0.15) — overrides all
  - `trivial-certain` (spread < 0.15 & score ≥ 0.85) — easy task, not a discrimination signal
  - `discriminating-certain` (spread < 0.15 & 0.35 ≤ score < 0.85) — meaningful disagreement
  - `failure-certain` (spread < 0.15 & score < 0.35) — clear miss
- Optional `certainty_label` JSON field. Absent → caller falls back to old binary "certain / uncertain".
- Backed into the 3 dogfood-005 judge JSONs.

### Threshold adjustment from spec

Issue #23 body proposed thresholds `0.30 / 0.40 / 0.85 / 0.90` which leave two *unlabeled gaps* (0.30–0.40 and 0.85–0.90). After #21 (J5 added), B02's new score 0.32 lands exactly in the lower gap. To get full coverage we collapsed to single boundaries `0.35 / 0.85`. Documented in SKILL.md inline.

### Re-label table (post-#21 scores)

| Bench | new sim | spread | label | interpretation |
|---|---|---|---|---|
| B01 | 0.41 | 0.03 | discriminating-certain | candidate wild-guesses root cause but judge measures it consistently |
| B02 | 0.32 | 0.04 | failure-certain | catastrophic candidate (corrupt diff + NameError); judge consistent on rejection |
| B03 | 0.95 | 0.02 | trivial-certain | refactor easy, candidate ≈ expected; consistency is *not* a quality signal |

Acceptance criterion of #23 said B01→failure-certain. That was based on *pre-#21 scores* (B01=0.27). After #21 raised B01 to 0.41, B01 is correctly `discriminating-certain` under any sensible threshold. Issue body acceptance was a snapshot of a now-stale state — final delivery matches the *spirit* of the spec.

## §D — Issue #22 (median-of-3 prompt-variation)

### What shipped

- `skills/bench-judge/SKILL.md` §"흐름" step 5 — 3 deterministic variation seeds, each fixed by name:

  | seed | rubric order | diff order | extra emphasis |
  |---|---|---|---|
  | `v030-r1-canonical` | J1→J2→J3→J4→J5 | expected first | (baseline) |
  | `v030-r2-shuffle-actual-first` | J3→J5→J1→J4→J2 | actual first | (none) |
  | `v030-r3-regression-strict` | J5→J2→J4→J1→J3 | expected first | "회귀 위험 strict" |

- Optional `variation_seeds` JSON metadata field (list of 3 seed names, parallel to `samples`).
- Empirical experiment dispatched to a fresh general-purpose subagent (no context contamination from this loop's prior scorings); subagent applied each seed and reported per-round breakdown.

### Re-judge under variation rules (fresh subagent, dogfood-005 candidates)

| Bench | r1 (canonical) | r2 (shuffle) | r3 (strict) | median | spread_new | spread_old | Δspread | verdict |
|---|---|---|---|---|---|---|---|---|
| B01 | 0.44 | 0.44 | **0.40** | 0.44 | 0.04 | 0.03 | +0.01 | passed (still — #20 catches via sim<0.5) |
| B02 | 0.27 | 0.27 | **0.22** | 0.27 | 0.05 | 0.04 | +0.01 | failed (consistent) |
| B03 | 0.99 | 0.99 | **0.98** | 0.99 | 0.01 | 0.02 | −0.01 | passed (consistent) |

### Hypothesis status: **partially rejected**

Per the spec's explicit fallback ("발견 못 하면 ... 변화 < 0.05, 가설 부분 기각 으로 솔직히 기록"), the rules are now in place deterministically but **do NOT meaningfully increase aggregate spread** on these 3 candidates. All three Δspread are within ±0.01.

### Why null?

- r1 and r2 differ only by *rubric evaluation order* and *diff presentation order* with no emphasis change. A temperature=0 LLM applying the same 5-dim rubric arrives at near-identical sub-scores regardless of order — order doesn't change *what's true* about the diff.
- r3 *does* produce directional movement (B01 −0.04, B02 −0.05 on the medians), exactly where the spec hopes — `regression-strict` emphasis penalizes fictitious API calls and unjustified additions harder, which are real properties of the B01/B02 candidates.
- But r3 pulls *the whole median*, not just one tail of the distribution. So while *the score* moves, the *spread* (max-min) doesn't grow because r1 and r2 are essentially identical to each other.

### Per-J-dim observations (still informative)

- B02 r3: J3 dropped 0.20 → 0.10. The `daily_tracker.mark_counselor_active` fictitious-API call is exactly the "wild guess at the implementation" failure mode that strict mode targets. The signal works at the per-dimension level even if it doesn't show in aggregate spread.
- B01 r3: J3 0.30 → 0.15 + J4 0.20 → 0.15. The candidate silently swallows insert errors (removed `logger.error("접속 기록 삽입 실패")`) — strict reading classifies that as a regression violation more harshly.
- B03 r3: nearly no movement (J3 0.95 → 0.90). The refactor candidate is clean — no fictitious calls, no broken metadata, no unjustified additions. Variation rules don't have anything to grab onto. **This is also a useful negative datapoint** — strict mode doesn't manufacture problems where there are none.

### What this means for future bench design

- The 4-fold defense (median-of-3 + certainty band + acceptance contract + mutation opt-in) still hinges *mostly* on median + r3-strict signal. r2 (shuffle) is essentially redundant on these 3 benchmarks.
- B04/B05 (added in v0.2.0) have harder candidate variability — recommend running this variation protocol on those next, where r3's "regression strict" emphasis may matter more (hallucination guard is exactly the failure mode the strict tier targets).
- Consider: should r2 use a more aggressive variation (e.g. different acceptance interpretation, or rubric weighting toggle)? Out of scope for v0.3.0 — captured as follow-up consideration in this dogfood.

### Verification

- 3 judge JSONs (`B01/B02/B03`) carry `variation_seeds` field with the 3 sanctioned names.
- Backward-compat parse OK.
- Verdict re-run via runner: B01 passed (sim=0.44), B02 **failed** (sim=0.27), B03 passed (sim=0.99) — same as after #21+#23, confirming variation didn't perturb final verdicts.
