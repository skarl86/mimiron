# 영업일 보정 없이 안내된 취소 마감일

## 증상

스마트스토어 채팅 봇이 "이용일 N일 전까지 취소 가능" 규정을 안내할 때,
*단순 달력 계산만 적용* 해서 결과가 주말·공휴일에 떨어져도 그대로
안내함. 결과적으로 사용자는 *실제 접수 불가한 날짜* 까지 취소 신청을
미루다가 마감 후 환불 거부 사례 발생.

예) 이용일이 2026-05-26(화) 이고 규정이 "이용일 3일 전까지" 인 경우
- 봇 안내: "5/23 까지 취소 가능"
- 실제: 5/23 은 토요일 → *직전 영업일* 5/22(금) 17:30 KST 가 진짜 마감

## 의도된 동작

- 마감일 계산은 결정론적 *코드* 가 담당하고, LLM 은 이용일·기준일수
  추출만 맡는다 (산수·요일 판정을 LLM 산수에 의존하지 않는다).
- 단순 달력 계산 결과가 주말·공휴일이면 *직전 영업일로 당김*.
- 영업일 판정 기준(평일 09:30~17:30 KST)은 한 곳에 정의된 단일
  진실 소스를 모든 호출처가 참조 (프롬프트·라우터·서비스에 같은
  업무시간을 직접 문자열로 적지 않는다).
- 한국 공휴일 데이터 출처는 외부 라이브러리·달력 등 결정론 데이터
  소스를 사용 (LLM 추론 결과로 휴일 여부를 판단하지 않는다).
- 취소·환불 의도뿐 아니라 *상품 문의* 의도에서도 동일 도구를 통해
  마감일을 안내한다 (양쪽 흐름이 같은 결과를 내야 한다).
- 응답 포맷은 보정 후 날짜를 *요일과 함께* 노출 (예: `2026-05-22(금)
  17:30 KST`).

## Acceptance criteria

- `agents/naver-smartstore/app/business_hours.py` 가 신설되어 업무시간
  상수(시작·종료 시각, 영업 요일 집합)와 표시용 포맷터를 노출한다.
- `agents/naver-smartstore/app/services/cancellation_deadline.py` 가
  신설되어 `compute_cancellation_deadline(tour_date, days_before)` 순수
  계산 함수와, 이를 감싸는 LLM 도구가 정의된다.
- `agents/naver-smartstore/app/utils.py` 에 `is_business_day` 와
  `last_business_day_on_or_before` 가 추가된다. 영업일 판정은 한국
  공휴일을 고려한다.
- `agents/naver-smartstore/app/routers/cancel_refund.py` 와
  `agents/naver-smartstore/app/routers/product.py` 양쪽에 마감일 계산
  도구가 등록되고, 해당 프롬프트(`cancel_refund.md`, `product.md`)에
  *직접 날짜 계산 금지·도구 호출 의무* 가 명시된다.
- 신설된 단위 테스트가 영업일 판정·마감일 보정(주말·공휴일·연말 경계)
  과 도구 라우팅을 잠근다.

## Out-of-scope

- 환불 처리 본 절차(접수 후 처리 로직) 는 손대지 않는다 — *마감일
  안내* 까지만 다룬다.
- 한국 외 지역(휴일 체계가 다른 지역) 대응은 별도 PR.
- 업무시간 외(주말·야간) 접수된 취소 요청을 *어떻게 큐잉할지* 는 본
  변경 범위 아님.
- 다른 에이전트(naver-talktalk, kakao 등) 의 동등 패턴 차용은 별도 PR.
